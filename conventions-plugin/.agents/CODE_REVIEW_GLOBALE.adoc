== 1. Famille 1 ‚Äî Test JUnit Configuration (present dans ~20 projets)

=== Pattern actuel

[source,kotlin]
----
dependencies {
    testImplementation(kotlin("test-junit5"))
    testRuntimeOnly("org.junit.platform:junit-platform-launcher")
}

tasks.withType<Test>().configureEach {
    useJUnitPlatform()
    testLogging {
        events("passed", "skipped", "failed")
        showStandardStreams = true
    }
}
----

=== Projets concernes

Tous sauf `intellij-plugins` (JetBrains Platform, pas Gradle). Soit ~20 projets.

=== Variantes identifiees

|===
| Variante | Projets
| `jvmArgs("-XX:+EnableDynamicAgentLoading")` ajoute | bakery, plantuml, codebase, capsule
| `maxHeapSize = "2g"` | codebase, plantuml
| `systemProperty("junit.jupiter.execution.parallel.enabled", "true")` | codebase seulement
|===

=== Convention plugin propose

`education.cccp.build.gradle-plugin` ‚Äî avec options :
- `enableDynamicAgentLoading` (default true)
- `maxHeapSize` (default null)
- `parallelExecution` (default false)

== 2. Famille 2 ‚Äî FunctionalTest Source Set (present dans ~12 projets)

=== Pattern actuel

[source,kotlin]
----
val functionalTest: SourceSet by sourceSets.creating {
    java { srcDirs("src/functionalTest/kotlin") }
    resources { srcDirs("src/functionalTest/resources") }
}

dependencies {
    add(functionalTest.implementationConfigurationName, gradleTestKit())
    add(functionalTest.implementationConfigurationName, kotlin("test-junit5"))
    add(functionalTest.implementationConfigurationName, libs.assertj.core)
    add(functionalTest.runtimeOnlyConfigurationName, "org.junit.platform:junit-platform-launcher")
}

val functionalTestTask = tasks.register<Test>("functionalTest") {
    testClassesDirs = functionalTest.output.classesDirs
    classpath = configurations[functionalTest.runtimeClasspathConfigurationName] + functionalTest.output
    useJUnitPlatform()
}

gradlePlugin.testSourceSets.add(functionalTest)

tasks.check { dependsOn(functionalTestTask) }
----

=== Projets concernes

bakery, plantuml, codebase, codex, capsule, readme, slider, hyperframes, dashboard, graphify, api-key-pool, planner

=== Variantes identifiees

|===
| Variante | Projets
| `configurations["functionalTestImplementation"].extendsFrom(‚Ä¶)` legacy | capsule, readme
| `testImplementation` extend `functionalTest` (reverse inheritance) | plantuml, readme
|===

=== Convention plugin propose

`education.cccp.build.functional-test` ‚Äî parametres :
- `sourceSetName` (default `functionalTest`)
- `additionalDependencies` (pour les depends specifiques projet)

== 3. Famille 3 ‚Äî Cucumber BDD Tests (present dans ~8 projets)

=== Pattern actuel

[source,kotlin]
----
sourceSets.test {
    resources.srcDir("src/test/features")
    java.srcDir("src/test/scenarios")
}

val cucumberTest = tasks.register<Test>("cucumberTest") {
    testClassesDirs = sourceSets.test.get().output.classesDirs
    classpath = configurations.testRuntimeClasspath.get() +
        sourceSets.test.get().output +
        sourceSets.main.get().output +
        files(tasks.jar.get().archiveFile)
    useJUnitPlatform { excludeEngines("junit-jupiter") }
    systemProperty("cucumber.junit-platform.naming-strategy", "long")
}

tasks.named<Test>("test") {
    filter { excludeTestsMatching("*.scenarios.*") }
}

tasks.check { dependsOn(cucumberTest) }
----

=== Projets concernes

codebase, bakery, plantuml, slider, readme, capsule, dashboard, agent-contracts (workspace-bom)

=== Variantes identifiees

|===
| Variante | Projets
| `cucumber.execution.parallel.enabled` | codebase
| Timeout 5min par scenario | bakery
| 32 cucumber tasks parametrees (data class CucumberTaskSpec) | codebase (unique, le plus avance)
| Plugins Cucumber (`io.cucumber:junit-platform-engine` 7.21.0) | tous
|===

=== Convention plugin propose

`education.cccp.build.cucumber` ‚Äî parametres :
- `featuresDir` (default `src/test/features`)
- `scenariosDir` (default `src/test/scenarios`)
- `parallel` (default false)
- `timeoutMinutes` (default null)
- Support multi-task pour le pattern avance codebase

== 5. Famille 5 ‚Äî Extension Class Patterns (2 styles concurrents, ~12 projets)

=== Pattern A ‚Äî Recommande (Gradle 8.x+)

[source,kotlin]
----
abstract class CodexExtension {
    abstract val zone: Property<LicenseZone>
    abstract val pgvectorHost: Property<String>
}
----

Utilise par : codex, planner, codebase (partiellement), dashboard, hyperframes, api-key-pool

=== Pattern B ‚Äî Legacy

[source,kotlin]
----
open class PlantumlExtension @Inject constructor(objects: ObjectFactory) {
    val configPath: Property<String> = objects.property(String::class.java)
}
----

Utilise par : plantuml, graphify, bakery, capsule, slider, newpipe

=== Recommandation

Normaliser vers **Pattern A** (`abstract class` + `abstract val`) pour tous les nouveaux developpements.
Migration progressive des projets legacy. Ce n'est pas un convention plugin ‚Äî c'est une **norme de code**
a documenter dans `AGENT.adoc` de chaque borough.

== Architecture Decision Records

=== ADR-001 ‚Äî Pourquoi des precompiled script plugins et pas un buildSrc partage ?

*Contexte* : Les precompiled script plugins (`.gradle.kts` dans `src/main/kotlin`) sont le
pattern recommande par Gradle pour les convention plugins. Alternatives : `buildSrc/` partage
(impossible cross-repo), `includeBuild()` (plus lourd), plugin binaire publie (trop tot).

*Decision* : Precompiled script plugins dans `conventions-plugin/src/main/kotlin/`,
references par `includeBuild()` depuis les projets consommateurs.

*Consequence* : Chaque projet consommateur doit ajouter `includeBuild("../conventions-gradle")`
dans son `settings.gradle.kts`.

=== ADR-002 ‚Äî Ordre de migration

1. Dashboard-gradle (projet nouveau, CNV-5 dogfood) ‚Äî valide les 3 patterns (nouveau projet)
2. Codebase-gradle (projet existant mature) ‚Äî valide la compatibilite ascendante
3. Capsule-gradle (projet legacy, patterns divergents) ‚Äî valide la migration des cas tordus
4. Generalisation aux 15+ autres projets
